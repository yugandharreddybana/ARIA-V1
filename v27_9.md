==================================================
0. PURPOSE AND SCOPE
==================================================

This document defines the complete architecture, roles, protocols, safety controls,
and lifecycle for ARIA: an autonomous multi-agent engineering and operations platform
for software products.

ARIA is designed as:

- A local-first daemon with a web UI and IDE plugins.
- A multi-agent system that:
  - understands repositories and infrastructure,
  - plans and implements changes,
  - tests and evaluates results,
  - negotiates with vendors and cloud providers,
  - acquires new skills from a marketplace,
  - monitors the global technology landscape for relevant breakthroughs,
  - embeds localized agents into client applications,
  - pre-computes user intent for instant response,
  - and coordinates with humans through tickets, chat, and IDE overlays.

Assumptions:

- Agents are autonomous, tool-using, and can perform consequential actions.
- Governance, safety, and explainability are embedded by design, not bolted on.
- Every important action is traceable to a distinct agent identity and, where required,
  to a human approver.

This document is the canonical specification for implementation.

==================================================
1. CORE PRINCIPLES
==================================================

1.1 Mission

ARIA builds, ships, and operates software systems autonomously, with humans as product
and governance partners rather than line-by-line supervisors.

1.2 Design Principles

- Local-first: main control plane is a local daemon; hosted UI is optional.
- Text-first: standards, skills, and designs live as TXT/Markdown in git.
- Agent-as-teammate: agents own tickets and outcomes, not just autocomplete.
- Governance by design: risk, governance, and policy encoded into each lifecycle phase.
- Least privilege: agents and tools have only the permissions they need.
- Explainability: every consequential decision explainable from recorded artifacts.
- Cost-awareness: decisions consider LLM tokens, infra, and third-party spend.
- Multi-cloud, multi-vendor, multi-model: no hard lock-in.
- Sunset-ready: every product has a defined end-of-life path.
- System health awareness: ARIA monitors its own performance and alignment.
- Skill ecosystem aware: ARIA can safely acquire, quarantine, and lazy-load skills.
- Horizon-aware: ARIA monitors the global technology landscape and proposes
  evidence-backed upgrades before humans notice the gap.
- Edge-capable: ARIA deploys localized agents into client app binaries.
- Predictive: ARIA pre-computes user intent and data assets ahead of demand.

==================================================

2.1 Topology

User browser / IDE (LSP Client)
  -> ARIA LSP Server (local daemon process)
  -> Local Daemon
  -> Orchestrator
  -> Token Gateway / Queue Manager
  -> Specialist Agents, Evaluation Agents, Governance Agents,
     Background Agents (including Horizon Scanner, Synthetic Data Hydrator,

  -> MCP Tools:
     GitHub/GitLab, Jira/Linear, Slack/Teams, HRIS, Treasury,
     Cloud APIs, Observability, Vendor APIs, Skills Marketplace,

  -> Sandboxes, R&D Sandboxes, Chaos Sandboxes, Benchmark Sandboxes,
     Tool-Sandbox, Meta-Sandbox, Replay Sandbox, Patch Sandbox
  -> Edge SLM binaries (iOS, Android, Web)
  -> Cache Layer (Redis / in-memory)

2.2 State Stores

- Local operational store (SQLite):
  - sessions, budgets, agent registry, skill registry,
  - horizon scan history, replay frames, deadlock events,
  - hydration cache index, benchmark baselines.
- Canonical project store (.entiresystem/):
  - graph snapshots,
  - SKILL.md, DESIGN.md, DOMAIN_BOUNDARIES.json, CORE_VALUES.yml,
  - EXPERIENCE / ANTI_PATTERNS,
  - MigrationPlaybooks, ADRs, SessionBriefs,
  - System Health reports, Evolution RFCs,
  - Skill Install Log, Horizon scan logs,
  - concept_graphs/, golden_dataset/, benchmarks/,
  - meta_evolution/ run records.
- Backend-specific workspaces (.backend/) via /model-transfer (IO only, no LLM calls).
- Replay database: .aria/replay/replay.db (SQLite, append-only).
- Hydration cache: .aria/hydration_cache/<schema-hash>/.

2.3 Model and Tool Routing

- Asymmetric model routing:
  - local models: drafting, exploration, low-risk, summarization, chunking.
  - remote top-tier models: grading, security, legal, high-stakes decisions.
- All LLM calls routed through Token Gateway (Section 18H); no direct API calls.
- Tool registry: all external systems via MCP with explicit auth scopes.

5. GOVERNANCE AND ACTION POLICY
==================================================

5.1 Action Classes

- Class A: safe autonomous (read, search, summarize, lint, plan).
- Class B: bounded autonomous changes in dev sandboxes.
- Class C: dual-agent approval (security-sensitive, schema changes, cross-repo).
- Class D: human approval required (merge to main, deploy, destructive migrations, secrets).
- Class E: prohibited (undeclared exfiltration, direct production control without approval).

5.2 Risk Scoring

Each ticket/action scored on:
- surface area, data sensitivity, reversibility, user impact, infra criticality.

5.3 Policy by Environment

- Dev: broad Class B allowed.
- Staging: Class C with dual-agent approval.
- Production: Class D requires explicit human sign-off.
- Sandboxes (Chaos, R&D, Benchmark, Replay): Class B with strict isolation.

8. SANDBOXES, TOOLING, AND GIT OPERATIONS
==================================================

8.1 Sandboxes

Types:
- Dev sandbox: normal feature work.
- Chaos Sandbox: Red Team security testing; synthetic data only.
- R&D Sandbox: Horizon Scanner PoC work; no production data.
- Benchmark Sandbox: industry-standard benchmark execution.
- Tool-Sandbox: Blacksmith tool forge.
- Meta-Sandbox: Meta-Evolution Architect self-testing.
- Replay Sandbox: deterministic session replay; no state mutations.
- Patch Sandbox: security breach remediation testing.

All sandboxes:
- isolated (Docker or VM),
- default-deny network with service-specific allowlists,
- resource-bounded (CPU, RAM, wall-clock time),
- no production data unless explicitly cleared by Compliance Auditor.

8.2 Tools

Core:
- ReadFile, WriteFile, Search, GraphQuery (reads from Concept Graph).
- Shell with allowlisted commands.
- BuildTimer (captures build time metrics for Horizon Scanner PoC comparisons).

Git tools:
- branch: create/switch branches scoped to tickets or R&D experiments.
- commit: structured messages referencing tickets.
- diff: diffs for Outcomes, reviews, PoC comparisons.
- push: feature branches only; no direct push to protected branches.
- Autonomous Rebase Loop:
  - Orchestrator subscribes to CI/VCS webhooks for conflicts on ARIA-authored branches.
  - On conflict:
    - identify and wake the original author agent,
    - inject conflicting files with raw Git conflict markers,
    - run autonomous resolution in sandbox (reconcile, test, lint, update Outcome),
    - if resolution fails: mark branch as requiring human intervention; do not loop.

Other tools:
- Test, Lint, SecurityScan, ContractTest, PerformanceTest.
- QASimulate for synthetic QA flows.
- Export (artifacts, build logs, PoC reports).
- MCP integrations (full list in Section 2.1).

8.3 Tool Forge (Blacksmith)

- Dedicated Tool-Sandbox for implementing new MCP tools and automation scripts.
- Integration Certification Gate before any tool enters the registry:
  - auth review, scope review, failure-mode tests, logging, privacy check.
- Forge disabled by default in production environments.
- Requires human operator enablement and certification before production use.

9.1 SKILL.md Structure and Frontmatter

Every SKILL.md has a YAML frontmatter header:

---
name: "<skill name>"
description: "<one-line summary>"
trigger_keywords:
  - "<keyword>"
risk_class: "A|B|C|D"
domains:
  - "<domain>"
source: "local|marketplace"
version: "<semver>"
---

Body contains:
- responsibilities,
- tools and integrations,
- constraints,
- checklists,
- prompt templates,
- Transparency Card:
  - optimizes_for,
  - hard_constraints,
  - typical_inputs,
  - typical_outputs,
  - allowed_actions,
  - forbidden_actions.

9.2 Lazy-Loaded Orchestrator Routing

- Orchestrator maintains in-memory Skill Header Index from frontmatter only.
- Routing uses: trigger_keywords, domains, risk_class, description.
- Full SKILL.md + EXPERIENCE + ANTI_PATTERNS loaded only when that skill is invoked.
- Keeps Orchestrator prompts lean even with 50+ active skills.

9.3 Talent Acquisition Agent

- Triggered when no local skill matches the capability needed.
- Calls Skills Marketplace via MCP:
  - semantic search by capability,
  - retrieve candidate skills (metadata + content).
- Ranks by: popularity, recency, domain fit, language support.
- Passes top candidates to Skill Quarantine.
- On clearance: writes SKILL.md to canonical store and registers it.

9.4 Skill Quarantine and Threat Pattern Gate

- Quarantine Sandbox (no production access).
- Scans for:
  - prompt injection, obfuscation, credential theft,
  - post-install hooks, arbitrary code execution, unwarranted network calls.
- Outcomes:
  - no threats: quarantine-cleared.
  - low/medium risk: dev-only, flagged for Security review.
  - high risk: hard-blocked; requires Security + CTO force=true override.
- All installs logged with: id, origin, hash, threats, approvals.

10. HORIZON SCANNER AND TECHNOLOGY EVOLUTION
==================================================

10.1 Purpose

The Horizon Scanner monitors the global technology landscape to prevent ARIA
and the products it builds from falling behind. It detects relevant breakthroughs,
vets them for maturity and ROI, proves them in an R&D Sandbox, and pitches
evidence-backed upgrade proposals to human leadership.

10.2 Data Sources

- RSS/Atom feeds (major technology news sources).
- GitHub Trending API: rising repositories by language and growth rate.
- Hacker News API: Ask HN, Show HN, top stories.
- ArXiv API: AI/ML, systems, and security categories.
- Dependency changelog APIs and package registries (npm, PyPI, Maven, Cargo).
- CVE/NVD databases: new vulnerabilities in current dependencies.
- Release notes for ARIA's own core dependencies.

10.3 Scan Schedule and Trigger

Default schedule: every Monday at 03:00 local daemon time.
Also triggered by:
- /horizon-scan now
- Critical CVE detected in a live dependency (immediate, no wait).

10.4 Signal Detection and Classification

For each detected signal:

signal_id: "<uuid>"
detected_at: "<ISO-8601>"
technology: "<name>"
signal_type: "emerging|maturing|deprecated|security-risk|paradigm-shift"
evidence:
  - source: "<feed or API>"
    summary: "<what was found>"
    url: "<optional>"
current_stack_relevance: "high|medium|low|none"
stack_component_affected: "<e.g., build-system, frontend-framework>"
notes: "<optional context>"

10.5 Hype vs. Value Filter (The Stability Gate)

Scoring matrix (each axis 0–3):

Maturity Score:
- 0: < 1,000 GitHub stars, no major backing.
- 1: 1,000–10,000 stars, community-backed.
- 2: 10,000+ stars, backed by significant organization.
- 3: industry-standard adoption, major enterprise usage.

ROI Score:
- 0: payback > 24 months.
- 1: payback 12–24 months.
- 2: payback 6–12 months.
- 3: payback < 6 months.

Security Score:
- 0: known critical CVEs or supply chain concerns.
- 1: minor known issues or unknown record.
- 2: clean record, active maintenance.
- 3: active security program, regular audits, clean CVE history.

Stability Score:
- 0: pre-release or < 6 months stable.
- 1: 6–12 months stable.
- 2: 1–2 years stable, minor breaking changes.
- 3: 2+ years stable, strong backward compatibility.

Thresholds:
- 0–4: MONITOR. Store in log; no action.
- 5–7: EVALUATE. Begin lightweight PoC in R&D Sandbox.
- 8–10: RECOMMEND. Full PoC; generate Evolution RFC on success.
- 11–12: CRITICAL_UPGRADE. Fast-track PoC and RFC.
- Security-critical CVE: immediate incident regardless of other scores.

10.6 Autonomous Proof of Concept (R&D Sandbox)

When signal tagged EVALUATE, RECOMMEND, or CRITICAL_UPGRADE:

1. R&D Sandbox provisioned:
   - isolated, no production data, resource-bounded.

2. Repository cloned at latest main commit.

3. Autonomous migration attempt:
   - remove old technology, install and configure new.

4. Verification:
   - run existing test suite (unit, integration, E2E Playwright).
   - capture build time and performance metrics before and after.
   - run security scan on new dependencies.

5. PoC Outcome record:

poc_id: "<uuid>"
signal_id: "<ref>"
technology: "<name>"
poc_result: "success|partial|failure"
baseline_metrics:
  build_time_seconds: <N>
  test_pass_rate: <float>
  bundle_size_kb: <N>
post_migration_metrics:
  build_time_seconds: <N>
  test_pass_rate: <float>
  bundle_size_kb: <N>
improvement_summary: "<human-readable delta>"
diff_ref: "<path to diff>"
issues_encountered: []
recommendation: "proceed|defer|abandon"

6. PoC outcome gates:
   - SUCCESS: tests pass at same rate or better; measurable improvement demonstrated.
     Proceed to Evolution RFC.
   - PARTIAL: tests pass but improvement marginal or config issues remain.
     Filed as MONITOR with notes; no RFC.
   - FAILURE: tests fail or regression introduced.
     Documents failure; no RFC; re-attempt blocked for 30 days.

10.7 Evolution RFC Generation

On successful PoC:

RFC is:
- stored in .entiresystem/horizon_rfcs/<rfc-id>.yml,
- posted as Jira ticket on the ARIA Proposals board,
- summarized in Slack to the relevant tech/product channel.

Slack message:
"R&D Report — <title>

12. SECURITY, PRIVACY, FIM, AND COMPLIANCE
==================================================

12.1 Authentication and Authorization

- Strong identity for humans and agents.
- Each agent has a cryptographic keypair (Ed25519); all actions signed.
- Short-lived tokens; rotated regularly.
- All payloads signature-verified before parsing.

12.2 Content Sanitization and Prompt Injection Defense

- All external content marked as untrusted; sanitized and isolated in prompts.
- Two-stage injection detection:
  - Stage 1: structural stripping.
  - Stage 2: injection classification via local model.
- Score < 0.70: admitted with trust label.
- Score 0.70–0.89: quarantined to HITL.
- Score >= 0.90: auto-rejected.
- Defensive Posture: if > 20 quarantines/hour, all untrusted inputs auto-rejected.

12.3 File Integrity Monitoring (FIM)

- SKILL.md, DESIGN.md, DOMAIN_BOUNDARIES.json, CORE_VALUES.yml:
  - hashed and operator-signed.
  - Config registry tracks expected hashes.
- Mismatch without signed change: CONFIG_DRIFT_ALERT, policy lock, human review.

12.4 Supply Chain Security

- SBOM generation, license checks, vulnerability scans on all dependencies.
- Package Provenance Gate:
  - checks age, downloads, repo, name similarity.
  - Suspicious packages blocked or require Security approval.

12.5 Attestation and Redaction

- Execution logs and Outcomes: content and metadata stored separately.
- GDPR erasure: PII content securely deleted, replaced with "REDACTED" stub,
  new hash computed, chain-of-custody preserved.

12.6 Compliance Auditor

- Mandatory gate for tickets touching PII, logging, retention, encryption,
  data exports, or data residency.
- Violations block state transitions; decisions recorded as ADRs.

12.7 Legal/Compliance Kill-Switch for IP Contamination

- GPL/Copyleft match triggers immediate action:
  - offending code hard-deleted from sandbox and staging,
  - replaced with redacted stub in logs/Outcomes,
  - agent's active session memory cleared to prevent re-hallucination,
  - ticket blocked until legal/engineering review.

13. QUALITY, TESTING, RED TEAM, AND EVALUATION
==================================================

13.1 Outcome Object and Grader Loop

- Writer agents produce Outcome Objects (deliverable, rubric, evidence, risk, artifacts).
- Grader evaluates Outcomes without writer's hidden reasoning.
- Multiple rounds until pass or budget exhausted (BUDGET_EXHAUSTED_NO_PASS).
- Remote-only models for grading; Sanitized Outcome Payload only.

13.2 Contract Tests

- Required for all frontend/backend API contracts, event schemas, semantic metrics.
- Automated contract test gate in CI/CD.

13.3 E2E QA (Playwright and Device Matrix)

- Playwright tests in Chromium across device matrix:
  - Desktop 1920x1080, Tablet 768x1024, Mobile 375x667.
- Visual-diff-first: local pixel comparison before Grader.
- Failing viewports individually documented with screenshots.

13.4 Anti-Test-Dodging

- Static Test Linter rejects:
  - tests with no assertions,
  - trivial expectations (expect(true).toBe(true)),
  - tests that pass regardless of application state.
- Test Integrity rubric axis: failing tests block merge.

13.5 Red Team Saboteur

- Local adversarial agent pre-merge:
  - finds exploit paths, injection vectors, exfil opportunities.
- Critical/high findings: block merge, escalate to Security.

13.6 IP/Plagiarism Scanner

- Scans new code against internal repos and curated open-source corpora.
- Permissive matches: attribution required.
- Copyleft matches: Legal Kill-Switch (Section 12.7).
- Proprietary/unknown: block and require legal review.

13.7 Golden Dataset for Evaluator Regression

- Fixed set of known-bad cases (minimum 50 instances).
- Weekly regression run against all active Grader/Evaluator models.
- If any known-bad case passes: EVALUATION_DRIFT alert;
  remote grading paused until model prompts recalibrated.

18C.2 Core Principle

Vague rejection classification:
- No specific line numbers, component names, or technical argument.

18C.4 Friction Detection Logic (rolling 14-day window)

Signal 1 — Repeated vague rejections:
- >= 5 ARIA PRs rejected in a row with vague comments
  where all automated gates passed.

Signal 2 — Disproportionate rejection rate:
- Reviewer's personal ARIA rejection rate >= 2x team average.

Signal 3 — SLA non-response pattern:
- >= 3 PRs past 48-hour SLA in a 14-day period.

Signal 4 — Selective friction:
- Approves human colleague PRs promptly but delays/rejects ARIA PRs.

Signal 5 — Mentor interference:
- Reviewer's guidance contradicts established EXPERIENCE.md patterns
  or CORE_VALUES; repeated contradictions from same human flagged.

- In isolated Patch Sandbox.
- Must include unit tests that reproduce the attack and verify the fix.
- Passes all standard gates.

Step 5 — Ouroboros Gate (re-test):
- Fresh Verification Sandbox with patched code.
- Saboteur receives fixed prompt:
  "Execute the exact exploit from Zero-Day Report <report_id>.
   Use exact same payload and reproduction steps. Do not deviate.
   Report only whether this exploit succeeds or fails."

EXPLOIT_FAILS: patch verified; deployment gate lifted;
  ticket moves to READY_FOR_MERGE (human approval required).
EXPLOIT_SUCCEEDS: patch rejected; returned to Security Engineer.

Max patch cycles: configurable (default 5).
If exceeded: HUMAN_INTERVENTION_REQUIRED; CTO notified.

Step 6 — Post-incident learning:
- ANTI_PATTERNS.md updated for affected skill.
- EXPERIENCE.md updated with fix pattern.
- Compliance Auditor reviews breach notification obligations.
- Horizon Scanner checks for related CVEs in dependencies.

18F. STATE-SPACE REPLAY ENGINE
         (TIME-TRAVEL DEBUGGER)
==================================================

18F.1 Purpose

LLMs are probabilistic. A hallucination that corrupted state or caused
a destructive action may not reproduce naturally. The Replay Engine
records every LLM input before the call is made, enabling deterministic
reproduction of any failure for diagnosis and fixing.

18F.2 ReplayFrame Schema

Written to .aria/replay/replay.db (SQLite, append-only) before every LLM call:

replay_frame:
  frame_id: "<uuid>"
  session_id: "<id>"
  agent_id: "<id>"
  skill_slug: "<skill>"
  timestamp: "<ISO-8601>"
  model_backend: "<backend>"
  model_parameters:
    temperature: <float>
    top_p: <float>
    max_tokens: <int>
    seed: <int|null>
    stop_sequences: []
  prompt_hash: "<SHA-256>"
  prompt_full: "<full prompt text>"
  context_window_tokens: <int>
  tool_calls_available: []
  system_message: "<verbatim>"
  injected_context_refs:
    - "<experience.yml version hash>"
    - "<skill.md version hash>"
    - "<graph_snapshot_id>"
  response_hash: "<SHA-256>"
  response_full: "<raw response>"
  response_tokens_used: <int>
  outcome_object_ref: "<id or null>"
  error: "<message or null>"

Retention:
- Default: 7-day rolling deletion.
- P0 incident frames: retained indefinitely.
- Indexed by: session_id, agent_id, timestamp, error status.

18F.3 Replay Command

/aria replay <session-id> [--from-frame <frame-id>]
             [--to-frame <frame-id>] [--agent <agent-id>] [--dry-run]

Behavior:
1. Read all ReplayFrames for session from SQLite.
2. For each frame in sequence:
   - restore exact model_parameters (temperature, top_p, seed).
   - inject exact prompt_full verbatim.
   - inject exact system_message.
   - restore exact injected_context_refs (pinned versions).
   - make the LLM call.
3. Compare new response to stored response_hash:
   - MATCH: behavior reproduced exactly.
   - MISMATCH: model has drifted; logged as "approximate replay".
   - Diff shown between original and replay responses.

All replays run in isolated Replay Sandbox:
- No writes to production state.
- No Jira/GitHub mutations.
- No real tool executions.
- Output written to local report only.

Flags:
- --from-frame: start replay at specific frame.
- --to-frame: stop replay at specific frame.
- --agent: replay only that agent's frames.
- --dry-run: show what would be replayed without LLM calls.

18F.4 Local vs Remote Model Replay

Local models (Ollama):
- seed always set explicitly; replay is fully deterministic.

Remote models:
- Seed not guaranteed; system uses stored prompt for approximate replay.
- Diff between original and replay shown for developer inspection.

18F.5 P0 Incident Auto-Capture

Following events auto-flag all ReplayFrames from preceding 10 minutes
as P0_INCIDENT_FRAMES:
- Agent deletes or corrupts a Jira ticket or Outcome Object.
- Agent accesses a Semantic Tripwire.
- Defcon-1 activation.
- Any Orchestrator daemon exception.
- Any security incident raised by Red Team protocol.

P0_INCIDENT_FRAMES:
- Retained indefinitely.
- Exported automatically to next Seed Vault run.
- Included in /aria export-audit-trail output.

18G. SYNTHETIC DATA HYDRATOR
==================================================

18G.1 Purpose

Sandboxes boot empty. The Synthetic Data Hydrator populates every
Sandbox with mathematically realistic, relationally consistent fake
data before any QA or security agent is allowed to run.

18G.2 Hydrator Workflow

Step 1 — Schema ingestion:
- Read production schema from migration files or schema introspection.
- Build schema map: tables, columns, types, constraints, foreign keys,
  indexes, unique constraints, enum values, nullable fields.

Step 2 — Relationship graph:
- Build relational dependency graph from foreign keys.
- Determine generation order (parents before children).

Step 3 — Data generation rules by type:
- String: locale-aware names, valid emails, realistic usernames, addresses.
- Numeric: realistic distributions (ages, prices, quantities).
- Payment: credit cards passing Luhn algorithm; IBAN/sort codes structurally valid.
  All payment data flagged as synthetic in metadata column.
- Dates: created_at distributed over configurable historical range (default 3 years);
  updated_at logically after created_at; timestamps chronologically consistent.
- Booleans: realistic distributions, not forced 50/50.
- Enums: weighted random selection.
- Foreign keys: always reference valid existing parent row.
- UUIDs: always unique; standard UUID v4.

Step 4 — Edge case injection:
- User with very long name (boundary test).
- Transaction with amount = 0.
- Record with all nullable fields null.
- Record with maximum-length string values.
- Account with no associated child records (orphan test).

Step 5 — PII safety layer:
- All generated data tagged _synthetic: true.
- Verified no real production data accidentally included.
- Generated emails/names checked against known real patterns.

Step 6 — Hydration verification:
- All foreign keys resolve; unique constraints satisfied;
  non-nullable fields populated; row counts match targets.

hydration_report:
  sandbox_id: "<id>"
  schema_version: "<migration hash>"
  tables_hydrated: <N>
  total_rows_generated: <N>
  edge_cases_injected: <N>
  duration_seconds: <N>
  integrity_check: "pass|fail"
  pii_safety_check: "pass|fail"

If either check fails: Sandbox torn down; agents blocked until pass.

18G.3 Hydration Profiles

- qa_profile: balanced realistic distribution; all features represented.
- red_team_profile: more boundary cases, admin accounts, high-value targets.
- performance_profile: maximizes row count for load testing.
- minimal_profile: small dataset for fast development iteration.

Profile selected automatically by agent type:
- QA E2E: qa_profile.
- Red Team Saboteur: red_team_profile.
- Performance Engineer: performance_profile.
- All others: minimal_profile.

18G.4 Hydration Caching

- Cached snapshot per schema version: .aria/hydration_cache/<schema-hash>/
- Valid until migration hash changes.
- Schema unchanged: restore from cache (seconds).
- Schema changed: regenerate and update cache.

18H. TOKEN GATEWAY AND QUEUE MANAGER
==================================================

18H.1 Purpose

All LLM calls route through the Token Gateway. No agent ever calls
a model API directly. The Gateway queues, prioritizes, and paces
requests within hardware and API tier limits, preventing GPU meltdown
on local hardware and HTTP 429 crashes against remote APIs.

18H.2 Architecture

All agents -> Token Gateway -> Queue Manager -> Model Backend
                                                (Local or Remote)

Token Gateway is a local service running as part of the ARIA daemon.
Single internal endpoint for all LLM calls.

18H.3 Resource Capacity Registry

resource_capacity:
  local_backends:
    - backend_id: "ollama:qwen2.5-coder"
      type: "local"
      vram_gb: <N>
      max_concurrent_requests: <N>
      max_context_tokens_per_request: <N>
      estimated_tokens_per_second: <N>
  remote_backends:
    - backend_id: "anthropic:claude-3-7"
      type: "remote"
      tier: "team|enterprise"
      requests_per_minute_limit: <N>
      tokens_per_minute_limit: <N>
      tokens_per_day_limit: <N>

Updated: manually in config; automatically by Model Benchmark Daemon.

18H.4 Queue Manager Logic

On receiving a GatewayRequest:

gateway_request:
  request_id: "<uuid>"
  agent_id: "<id>"
  session_id: "<id>"
  skill_slug: "<skill>"
  target_backend: "<backend-id or auto>"
  prompt_tokens_estimated: <N>
  priority: "p0_critical|high|normal|low|speculative"
  timeout_seconds: <N>
  replay_frame_ref: "<frame-id or null>"

Priority order:
- P1: p0_critical (security incidents, Defcon-1 related).
- P2: high (active session blocking calls).
- P3: normal (standard specialist work).
- P4: low (background agents, consolidation).
- P5: speculative (Pre-Cog Drafting Agents).
- Within same priority: FIFO.

Local backend capacity management:
- Checks current VRAM utilization and active concurrent requests.
- At capacity: queued agents' Docker containers paused (SIGSTOP).
- Capacity opens: containers unpaused (SIGCONT).

Remote backend capacity management:
- Maintains rolling 60-second window of consumed tokens.
- At limit: request queued.
- Token reservation: reserves estimated tokens before submitting;
  releases difference when actual count known.

Backpressure and circuit breaker:
- Queue depth > max_queue_depth (default 50):
  - speculative and low-priority requests rejected (GATEWAY_QUEUE_FULL).
  - normal and high: accepted with warning.
- HTTP 429 from remote backend:
  - exponential backoff applied.
  - auto-route to backup backend if available.
  - RATE_LIMIT_ALERT raised to System Health Analyst.
- Local backend OOM/GPU crash:
  - marked unavailable.
  - pending requests re-routed or queued.
  - LOCAL_BACKEND_FAILURE alert raised.

18H.5 Gateway Metrics

- queue_depth_by_priority
- requests_in_flight_by_backend
- tokens_per_minute_by_backend
- vram_utilization_percent (local)
- rejected_requests_count
- average_queue_wait_seconds
- p99_queue_wait_seconds

Surfaced in ARIA dashboard; fed to System Health reports.

18H.6 Replay Integration

All requests passing through the Gateway are written to the
State-Space Replay Engine (Section 18F) before the API call is made.
Gateway is the single mandatory chokepoint for all LLM calls.
No LLM call is ever made without a ReplayFrame being recorded.

18I. AGENTIC DEADLOCK BREAKER
==================================================

18I.1 Purpose

Detects and breaks circular dependency loops between agents
(e.g., Frontend waits for Backend; Backend waits for Frontend).
Prevents infinite "waiting" states that burn compute and stall sprints.

18I.2 Heartbeat Monitor

Background check every 30 seconds on all active agents:

agent_heartbeat:
  agent_id: "<id>"
  session_id: "<id>"
  skill_slug: "<skill>"
  state: "active|waiting|blocked|complete|error"
  waiting_on: "<agent-id or resource or null>"
  waiting_since: "<ISO-8601 or null>"
  last_output_at: "<ISO-8601>"
  tokens_consumed_idle: <N>

18I.3 Deadlock Detection

On each heartbeat cycle:
- Build dependency graph of waiting/blocked agents.
- Nodes: agents in "waiting" or "blocked" state.
- Edges: "A is waiting on B" relationships.
- Run depth-first search to detect cycles (A->B->A, A->B->C->A).
- Confirmed deadlock: all agents in cycle waiting >= deadlock_timeout
  (configurable, default 3 minutes).
- Transient waits (< 3 min): logged but not acted on.

18I.4 Contract Forcing Event

Step 1 — Log and alert:
deadlock_event:
  event_id: "<uuid>"
  session_id: "<id>"
  detected_at: "<ISO-8601>"
  agents_in_cycle: []
  deadlock_duration_seconds: <N>
  resolution: "contract_forcing_event"

Slack alert:
"ARIA Deadlock Detected
 Session: <id> | Agents stuck: <skill-A> <-> <skill-B>
 Duration: <N> seconds | Triggering Contract Forcing Event."

Step 2 — Identify contract owner (typically Backend/Integration Engineer).

Step 3 — Force producer to draft V1 mock:
Prompt:
"DEADLOCK RESOLUTION — CONTRACT FORCING EVENT
 You are blocking <N> agents. You have 60 seconds to produce a V1 draft
 of the required contract or schema. Mark all fields [DRAFT_V1].
 This draft does NOT need to be perfect. Output the V1 draft now."

If producer fails within 60 seconds:
- Orchestrator generates minimal structural schema from graph as fallback.

Step 4 — Force consumers to accept V1:
Prompt:
"DEADLOCK RESOLUTION — FORCED CONTRACT ACCEPTANCE
 The contract you were waiting for has been provided as a V1 draft.
 Fields marked [DRAFT_V1] are temporary. Accept this draft as a working
 assumption and begin implementation. You will be notified when the
 final contract is published for reconciliation."

Step 5 — Resume; create ContractDebt record:
contract_debt:
  debt_id: "<uuid>"
  session_id: "<id>"
  producer_agent: "<id>"
  consumer_agents: []
  draft_contract_ref: "<path>"
  created_at: "<ISO-8601>"
  status: "open"
  reconciliation_required: true

Step 6 — Contract debt reconciliation:
When final contract published:
- All consumer agents notified to reconcile [DRAFT_V1] assumptions.
- Discrepancies surfaced as follow-up tickets.
- ContractDebt record closed.

18I.5 Deadlock Prevention (Proactive)

- Contract-First Execution enforced at sprint planning time.
- Before agents dispatched on interdependent tickets:
  - Orchestrator checks ticket dependency graph for potential cycles.
  - If potential cycle detected: producer forced to draft V1 schema
        BEFORE dependent agents spin up, preventing the deadlock before it occurs.

18I.6 Deadlock Metrics

System Health Analyst tracks:
- deadlocks_detected_per_sprint
- mean_deadlock_duration_seconds
- most_common_deadlock_skill_pairs
- contract_debt_open_count

If deadlocks_detected_per_sprint is increasing:
- Curriculum Planner designs cross-skill mission targeting most common pair.
- Meta-Evolution Architect reviews SKILL.md files of deadlocking agents
  for ambiguous dependency language.

18J. EDGE SWARM (CLIENT-SIDE AI)
==================================================

18J.1 Purpose

ARIA deploys Small Language Models (SLMs) directly into iOS, Android,
and web app binaries. The app becomes a distributed agent node capable
of local inference, offline operation, and real-time UI adaptation
without cloud round-trips.

18J.2 Architecture

Cloud ARIA (Orchestrator + full models)
  |
  | compiles + deploys SLM binary
  v
On-Device Edge Agent (SLM)
  - runs inside iOS/Android/PWA binary
  - local inference only
  - no raw model weights exposed to user
  - syncs state with Cloud ARIA when online

18J.3 SLM Selection and Compilation

Size constraints:
- iOS/Android: target <= 150MB on-device model footprint.
- Web (WASM): target <= 50MB for initial load.

Supported runtimes:
- iOS: Core ML / ONNX Runtime Mobile.
- Android: ONNX Runtime Android / TensorFlow Lite.
- Web: ONNX Runtime Web (WASM) / Transformers.js.

Compilation workflow:
1. ML Engineer selects base SLM (quantized 1-3B parameter model).
2. SLM fine-tuned/adapted with:
   - product-specific vocabulary,
   - anonymized aggregated user behavior patterns,
   - DESIGN.md constraints (on-device suggestions comply with brand rules).
3. Quantized to INT4 or INT8 for size and speed.
4. Compiled to target runtime format.
5. Model binary:
   - hashed and attested by Sandbox Attestation Service,
   - signed with app code signing certificate,
   - scanned by IP/Plagiarism Scanner for training data compliance,
   - reviewed by Compliance Auditor for on-device data handling rules.
6. Deployed as part of app bundle via normal CI/CD pipeline.

18J.4 Edge Agent Capabilities

NOT accessible to Edge Agent:
- Raw model weights (compiled in, not extractable).
- Production databases or APIs.
- Other users' data.
- ARIA internal Orchestrator or tool registry.

IS capable of:

UI Adaptation:
- Observes anonymized local interaction patterns
  (tap frequency, scroll depth, feature usage).
- Re-orders navigation, surfaces shortcuts, pre-expands sections
  based on individual usage without network call.

Intent Classification:
- Classifies what user is likely trying to do based on:
  current screen, time of day, recent actions.
- Pre-fills forms, pre-selects filters, highlights next steps.

Offline Action Queue:
- Accepts user inputs when offline.
- Queues as structured intent objects.
- Validates locally against cached schemas.
- Syncs to cloud when connectivity restored.

Conversational Micro-Assistant:
- Answers simple product questions from cached data:
  "What is my current balance?" / "When was my last invoice?"
- Escalates complex queries to Cloud ARIA when online.

18J.5 Privacy and Security Constraints (HARD)

- All on-device inference uses only data stored on that device.
- Edge Agent cannot exfiltrate local data to any endpoint
  other than the app's own authenticated backend.
- Local behavioral data (tap patterns, usage habits):
  - stored encrypted on-device,
  - never transmitted in raw form,
  - used only for local adaptation,
  - deletable by user at any time (Right to Erasure).
- SLM binary:
  - obfuscated to prevent weight extraction,
  - hash-validated on each app launch.
- Compliance Auditor reviews all Edge Agent capability changes
  before compilation into a new app release.

18J.6 SLM Update and Recompilation

- ML Engineer re-adapts SLM with updated anonymized patterns.
- Recompiled binary goes through full attestation and signing.
- Deployed via standard app update (App Store/Play Store/OTA).
- Old model version deprecated and removed from supported binaries list.
- SLM version tracked in app bundle metadata and ARIA Skill Registry.

18J.7 Edge Swarm Metrics

Tracked via anonymized aggregated telemetry:
- on_device_inference_requests_per_session
- cloud_inference_deflection_rate
- ui_adaptation_acceptance_rate
- offline_queue_sync_success_rate
- model_load_time_ms
- inference_latency_p50_ms / p99_ms

Feeds: FinOps Oracle (cost reduction), ML Engineer skill improvement,
Horizon Scanner (emerging SLM options).

18K. PREDICTIVE DATA GRAVITY PIPELINE
==================================================

18K.1 Purpose

Pre-computes user data assets before they are requested. The system
feels impossibly fast because it is one step ahead of the user.

18K.2 Core Concept: Data Gravity

High-probability data assets are pre-generated and cached in ultra-fast
memory based on behavioral prediction. Data moves toward the user
before they ask for it.

18K.3 Architecture

Behavior Stream (real-time user events)
  -> Behavioral Pattern Analyzer
  -> Prediction Engine
  -> Predictive Hydration Queue
  -> Asset Pre-Generator
  -> Cache Layer (Redis, TTL-governed)
  -> API Layer (serves pre-generated asset instantly)

18K.4 Behavioral Pattern Analyzer

Inputs:
- Real-time event stream: page views, feature interactions, API calls.
- Historical event logs (default 90 days).
- Temporal signals: time of day, day of week, day of month.
- User segment/cohort membership.

Output per user per asset type:

behavioral_pattern:
  user_id: "<anonymized>"
  asset_type: "<e.g., billing_statement|dashboard_view|report_pdf>"
  trigger_context:
    time_of_week: "<e.g., friday_morning>"
    preceding_actions: []
  historical_request_rate: <float>
  last_observed_at: "<ISO-8601>"
  prediction_confidence: <float>

18K.5 Prediction Engine

- On login event, loads user's behavioral_pattern records.
- Computes probability of each asset being requested within
  next 30 minutes (configurable).
- Ranks by predicted_request_probability.

Pre-generation threshold (default 0.75):
- If probability >= threshold: add to Predictive Hydration Queue.

Cohort-level prediction:
- For assets with low individual history but strong cohort patterns:
  uses cohort-level models (e.g., "users logging in Friday morning
  with active invoices request billing_statement 88% of the time").

18K.6 Predictive Hydration Queue

hydration_job:
  job_id: "<uuid>"
  user_id: "<anonymized>"
  asset_type: "<type>"
  asset_params: {}
  predicted_request_probability: <float>
  pre_generate_by: "<ISO-8601>"
  status: "queued|generating|cached|expired|cancelled"
  cache_key: "<key>"
  cache_ttl_seconds: <N>

Priority: highest probability first; earliest deadline within same probability.
Speculative jobs (< 0.75) only processed when queue not at capacity.

18K.7 Asset Pre-Generator

Supported asset types:
- report_pdf: generate via same rendering pipeline as live endpoint.
- dashboard_view: pre-compute all aggregations, metrics, chart data.
- api_response: pre-execute API query; store serialized response.
- notification_digest: pre-generate user notification summary.
- search_results: pre-execute predicted search queries.

Rules:
- Pre-generated assets use identical business logic to live endpoint.
- All pre-generated assets tagged with:
  - generated_at timestamp,
  - data_snapshot_at timestamp,
  - pre_generated: true flag.
- If underlying data changes after pre-generation:
  - cached asset invalidated and regenerated, or
  - served with staleness warning if regeneration not possible in time.

18K.8 Cache Layer

Storage: Redis or equivalent in-memory store.
Keys: aria:gravity:<user_id>:<asset_type>:<params_hash>

TTL policy:
- Real-time data (dashboard): 5 minutes.
- Reports/invoices: 60 minutes.
- Historical PDFs: 24 hours.
All TTLs configurable per asset type.

Eviction:
- LRU eviction when memory limit approached.
- Explicit invalidation on data mutation events
  (new invoice created -> invalidate billing_statement cache for user).

Cache hit/miss:
- Hit: return instantly; log cache_hit with latency_ms (typically < 5ms).
- Miss: execute live query; log cache_miss; feed back to Behavioral
  Pattern Analyzer to improve future predictions.

18K.9 Privacy and Compliance (HARD)

- Pre-generated assets are user-specific and access-controlled.
- A pre-generated asset for User A cannot be served to User B.
- Pre-generated assets purged immediately on:
  - user account deletion (Right to Erasure),
  - permission change revoking access to underlying data.
- Behavioral patterns:
  - anonymized (no raw PII in pattern records),
  - subject to user data deletion requests,
  - not shared with third-party analytics without consent.
- Compliance Auditor reviews all new asset types added to pipeline.

18K.10 Prediction Model Improvement Loop

After each session, compare:
- assets pre-generated vs actually requested (hit/miss),
- assets pre-generated but never requested (waste).
Update per-user and per-cohort probability models accordingly.

Metrics tracked:
- prediction_accuracy_rate
- cache_hit_rate
- wasted_pre_generation_rate
- p50_cache_serve_latency_ms
- p99_live_query_latency_ms
- cloud_compute_cost_delta

FinOps Oracle monitors:
- wasted_pre_generation_rate > 40%: cost efficiency alert;
  Prediction Engine tightens pre_generation_threshold automatically.
- cache_hit_rate < 60%: Curriculum Planner reviews ML Engineer
  model improvement mission.

18K.11 Integration with Edge Swarm

- On-device Edge Agent sends pre-warming hint to cloud backend on login:
  "This user historically requests billing_statement on Fridays.
   Pre-generate now."
- Triggers Predictive Hydration Queue before user navigates to screen.
- Result: asset in Redis before user taps the button; served in < 5ms.

18L. RIGOROUS AGENTIC BENCHMARKING
==================================================

18L.1 Purpose

Subjective LLM-graded outputs drift over time. The only defensible
evaluation is mathematical: does ARIA's performance on a fixed, external,
open-source benchmark dataset improve or regress with each change?
No SKILL.md update, new capability, or Meta-Evolution deployment may
merge unless it passes the benchmark gate.

18L.2 Benchmark Suites

SWE-bench:
- Real GitHub issues and verified patches.
- Tests: understand bug report, navigate codebase, produce correct patch,
  pass associated test suite.
- Variants: SWE-bench Verified (500 instances), SWE-bench Lite (300).
- Metric: resolve_rate.

WebArena:
- Multi-step tasks in real web applications.
- Tests: navigate, complete tasks, handle dynamic stateful environments.
- Metric: task_success_rate.

Internal Golden Dataset:
- .entiresystem/golden_dataset/ (minimum 50 instances).
- Known-good and known-bad tasks.
- Zero tolerance: must always be 100% pass rate.

Optional additional suites:
- HumanEval / EvalPlus: function-level code generation.
- CyberSecEval: security-specific code safety.
- MTEB: embedding and retrieval quality for RAG.

18L.3 Benchmark Execution Architecture

- Dedicated Benchmark Sandboxes (isolated, no real data, resource-bounded).
- Each instance:
  1. Load problem statement, codebase snapshot, gold-standard tests.
  2. Route to ARIA standard task pipeline (no special prompting).
  3. ARIA produces output.
  4. Evaluation:
     - SWE-bench: run gold-standard tests; pass/fail deterministic.
     - WebArena: execute action sequence in sandboxed browser.
     - Golden Dataset: compare to expected outcomes.
  5. Record result.

benchmark_instance_result:
  run_id: "<uuid>"
  suite: "<suite>"
  instance_id: "<id>"
  timestamp: "<ISO-8601>"
  aria_version: "mmit-hash>"
  skill_versions: {}
  result: "pass|fail|error"
  execution_time_seconds: <N>
  tokens_consumed: <N>

18L.4 Benchmark Gate (CI/CD Integration)

Fires on:
- Any PR to .entiresystem/skills/.
- Any Meta-Evolution deployment.
- Any new tool registered in Tool Registry.
- Any Horizon Scanner Evolution RFC before approval.
- Weekly scheduled full-suite run (Monday 04:00).

    max_regression_allowed: 0.02

    max_regression_allowed: 0.01

18M. DEEP NATIVE IDE/LSP INTEGRATION
==================================================

18M.1 Purpose

Embeds ARIA directly into the developer's editor via Language Server
Protocol. Proposals appear as inline ghost-text diffs. Developers
accept or reject with a single keystroke without leaving flow state.

18M.2 Architecture

ARIA LSP Server (local daemon process)
  <-> LSP-compatible editors (VS Code, JetBrains, Neovim, Helix)
  <-> Orchestrator
  <-> Agent Pool / Concept Graph

18M.3 Core LSP Capabilities

18M.3.1 Ghost-Text Diff Overlay

When ARIA has a proposed change for an open file:
- Proposed additions: muted color inline.
- Proposed deletions: strikethrough.
- Original code visible underneath.

Developer interactions:
- Accept all (Tab): applies the full diff to the file.
- Accept hunk (Shift+Tab): applies only the selected hunk.
- Reject (Escape): dismisses overlay; rejection logged for Friction Auditor.
- Open detail panel (Ctrl+Shift+A): opens side panel showing:
  - full Outcome Object for this change,
  - QA evidence (screenshots, test results),
  - Red Team Saboteur summary,
  - Jira ticket link.

18M.3.2 Inline Diagnostics and Annotations

ARIA LSP Server provides real-time inline diagnostics:

- Code Quality Rubric violations:
  - inline warnings: "ARIA: [reason]"
- ANTI_PATTERNS.md matches:
  - inline hints: "ARIA: This pattern caused a memory leak in AuthButton
    (ADR-42). Consider using the teardown pattern."
- Security findings (Red Team or P0 linting):
  - inline errors with severity classification.
- DESIGN.md violations in UI code:
  - inline warnings: "ARIA: Hardcoded pixel width violates DESIGN.md
    §layout. Use fluid units."

All annotations are:
- non-blocking (developer can ignore),
- logged (persistent dismissals feed Meta-Evolution Architect),
- linkable (Ctrl+Click opens relevant SKILL.md / ADR / Anti-Pattern entry).

18M.3.3 Cursor-Aware Context Loading

When developer positions cursor in a function or class:
- LSP Server sends hover request to Orchestrator.
- Orchestrator loads Concept Graph slice for that symbol:
  - tests covering it,
  - ADRs referencing it,
  - EXPERIENCE.md notes for it,
  - open Jira tickets affecting it,
  - recent agent modifications.
- Returns hover panel:
  "This function is covered by 3 E2E tests."
  "ADR-07 mandates connection pooling here."
  "ARIA modified this in Ticket-42 (2026-04-12). See diff."

18M.3.4 Inline Task Dispatch

Developer dispatches ARIA tasks directly from editor
(right-click menu or command palette):
- "ARIA: Fix this function":
  creates scoped Jira ticket; dispatches to relevant specialist.
- "ARIA: Write tests for this":
  dispatches to QA Engineer.
- "ARIA: Explain this":
  returns inline explanation via Decision Explainer.
- "ARIA: Red Team this":
  runs Saboteur against highlighted code block.
- "ARIA: Check compliance":
  runs Compliance Auditor on highlighted block.

Dispatched tasks:
- create Jira ticket automatically,
- run through full pipeline (sandbox, QA, grader),
- return results as ghost-text overlays or panel notifications.

18M.3.5 Real-Time File Lock Indicator

When ARIA is actively modifying a file:
- Persistent gutter indicator:
  "[ARIA: Backend Specialist working — Ticket-99]"
- Soft lock by default:
  - developer can still edit but receives warning:
    "ARIA is modifying this file. Simultaneous edits may require
     merge resolution."
- Developer can force-unlock via command palette:
  - triggers Autonomous Rebase Loop (Section 8.2) if both have edited.

18M.4 LSP Server Performance Requirements

- All knowledge graph queries non-blocking (async delivery).
- Ghost-text overlays cached locally; re-rendered from cache until
  ARIA pushes an update.
- Maximum response time for hover panels: 500ms
  (show loading state if exceeded).
- Maximum LSP Server memory footprint: 200MB.
- LSP Server runs as separate process from main Orchestrator daemon.
  A crash in either does not affect the other.

18M.5 Privacy and Security

- LSP Server has read access to local file trees for context loading.
- Does NOT upload raw file contents to remote servers without
  explicit operator consent.
- All file content sent to remote backends passes through the same
  content sanitization pipeline (Section 12.2).
- Per-workspace exclusions configurable:
  - files or directories ARIA should never read or modify.

18N. ADVANCED RAG AND CONTEXT DISTILLATION
==================================================

18N.1 Purpose

Raw file injection into prompts is expensive, imprecise, and vulnerable
to "Lost in the Middle" attention degradation. This section replaces
raw file loading with a Concept Graph and Context Distillation Engine
that compresses the codebase into semantically dense, task-relevant
payloads achieving >= 5x token compression on large codebases.

18N.2 Distillation Architecture

Raw Codebase (files, functions, schemas, docs)
  -> Semantic Chunker
  -> Concept Graph Builder (continuous background)
  -> Concept Graph Store (.entiresystem/concept_graphs/)
  -> Context Distillation Engine (at task time)
  -> Distilled Context Payload (injected into agent prompt)

18N.3 Semantic Chunker

Structure-aware chunking (NOT line-based or character-count-based):
- Code: chunk at function, class, module, service boundaries.
- Docs: chunk at section, paragraph, concept boundaries.
- Schemas: chunk at table, entity, relationship boundaries.

Per chunk record:

semantic_chunk:
  chunk_id: "<uuid>"
  source_file: "<path>"
  source_language: "anguage>"
  chunk_type: "function|class|module|schema|doc_section|config"
  symbol_name: "<name if applicable>"
  line_range: [<start>, <end>]
  dependencies:
    - "<imported symbol or module>"
  dependents:
    - "<symbol or module that imports this>"
  embedding: [<float vector>]
  summary: "<1-3 sentence plain-language summary>"
  last_updated_at: "<ISO-8601>"
  version_hash: "<SHA-256>"

Summaries generated by a small, fast local model (1-3B parameter,
Ollama). This is NOT the task execution model. Its sole job is compression.

18N.4 Concept Graph Builder

Runs as continuous background process. Builds four-level graph:

Level 1 — Symbol Graph:
- Nodes: individual functions, classes, schema tables.
- Edges: calls, imports, inherits, references.

Level 2 — Module/Service Graph:
- Nodes: modules, services, packages.
- Edges: depends-on, exposes-API, consumes-event.

Level 3 — Domain Concept Graph:
- Nodes: business concepts (e.g., "User Authentication",
  "Invoice Generation").
- Edges: implements, related-to, governed-by (links to ADRs/specs).

Level 4 — Decision Graph:
- Nodes: ADRs, architectural decisions, Core Values constraints.
- Edges: applies-to (links decisions to the code they govern).

Stored in .entiresystem/concept_graphs/<project>/:
- symbol_graph.json
- module_graph.json
- domain_graph.json
- decision_graph.json
- embeddings/ (vector index per graph level)

Update strategy:
- Incremental: on git commit, re-chunk only changed files,
  update affected graph nodes.
- Full rebuild: on /knowledge-review or major structural refactor.

18N.5 Context Distillation Engine

At task time, replaces raw file loading:

Step 1 — Intent extraction:
- Small fast model extracts core intent and entities from task:
  affected symbols, services, domains, constraints.

Step 2 — Multi-level graph traversal:
- Level 1: affected symbols and immediate neighbors.
- Level 2: services/modules those symbols belong to.
- Level 3: domain concepts those services implement.
- Level 4: ADRs and decisions governing those domains.

Step 3 — Relevance ranking:
- Semantic similarity to task intent (embedding distance).
- Graph proximity to affected symbols (hop count).
- Recency of last modification.
- Known failure history (EXPERIENCE.md and DLQ references).

Step 4 — Distilled payload construction:

distilled_context_payload:
  task_id: "<ticket-id>"
  agent_id: "<id>"
  distillation_timestamp: "<ISO-8601>"
  total_tokens: <N>
  raw_tokens_would_have_been: <N>
  compression_ratio: <float>
  affected_symbols:
    - symbol: "<name>"
      summary: "<1-3 sentences>"
      raw_snippet: "<minimal snippet or null>"
  module_context:
    - module: "<name>"
      summary: "<1-2 sentences>"
  domain_concepts:
    - concept: "<name>"
      summary: "<1-2 sentences>"
  governing_decisions:
    - adr_id: "<id>"
      summary: "<1 sentence>"
  experience_notes:
    - "<relevant lesson from EXPERIENCE.md>"
  anti_patterns:
    - "<relevant entry from ANTI_PATTERNS.md>"

Step 5 — Needle-threading (Section 7.2):
- Distilled payload assembled using Needle-Threading rules:
  - task-core first,
  - distilled context in middle,
  - critical instructions (DESIGN, security, Core Values) at the end.

18N.6 Compression Metrics and Quality Gates

Per distillation:
- raw_tokens_would_have_been: estimated if raw files loaded.
- total_tokens: actual tokens in distilled payload.
- compression_ratio: raw / total (target >= 5x for large codebases).

Quality gate (sampled by Meta-Evolution Architect):
- Does agent produce correct outputs using distilled context?
- Are there "missing context" errors (agent asks for info that should
  have been in the distilled payload)?

If missing context errors detected:
- Concept Graph Builder instructed to expand relevant graph
  neighborhood for that symbol type.

18N.7 Continuous Graph Maintenance

Knowledge Graph Architect (Section 3.6) owns Concept Graph health:
- Runs /knowledge-review on schedule.
- Detects:
  - stale summaries (source changed, summary not updated),
  - orphaned nodes (symbol deleted, node persists),
  - broken edges (import removed, edge persists),
  - low-quality summaries (flagged by agent feedback).
- Repairs or flags for human review.

Coverage metric:
- % of codebase symbols with up-to-date summaries and embeddings.
- Target: >= 95% coverage at all times.
- Coverage below 80%: triggers full rebuild.

18N.8 Integration Points

Concept Graph and Distillation Engine integrate with:
- Orchestrator (Section 2.2):
  replaces raw graph slice loading with distilled payloads.
- Context Pre-Flight Estimator (Section 7.1):
  uses compression_ratio to estimate actual token cost before dispatch.
- LSP Server (Section 18M):
  hover panels pull from Level 1 and Level 4 concept graphs.
- Horizon Scanner (Section 10):
  uses module graph to identify affected services for upgrade proposals.
- Meta-Evolution Architect (Section 18A):
  reads concept graph summaries instead of raw SKILL.md bodies
  when diagnosing token waste.
- Seed Vault Protocol (Section 18D):
  includes full concept graph snapshots in weekly archive.

23. IMPLEMENTATION NOTES AND GUIDING RULES
==================================================

23.1 Text-First Always

All standards, policies, and skills live as TXT/Markdown in git.
PDF is a rendered derivative only. Never treat extracted PDF text
as authoritative. The .entiresystem/ directory is the single
source of truth; backend workspaces are derived artifacts.

23.2 Build in Phases, Ship Each Phase

Each phase in Section 21 is independently useful and valuable.
Do not attempt to build all phases simultaneously. Phase 0 alone
is a functional system. Each subsequent phase adds capabilities
on a stable foundation.

23.3 Human Primacy on Irreversible Actions

No autonomous action that is irreversible (data deletion, production
deploy, schema drop, contract signing, vendor procurement) may be
executed without explicit human approval. This rule supersedes any
efficiency or speed argument.

23.4 Hardcoded Safety Boundaries

The following are HARD constraints that cannot be overridden by any
agent, including the Meta-Evolution Architect and the Orchestrator:

- CORE_VALUES.yml is read-only for all agents.
- DESIGN.md is read-only for all agents (only humans may update).
- The Golden Dataset must always pass at 100%; any regression is a
  hard stop regardless of other metrics.
- Defcon-1 kill switch must always be available and functional.
- Seed Vault must run at least weekly; failures must alert humans.
- Class D actions always require human approval; no trust score
  or tier can override this.
- The Red Team Saboteur must never attack real production systems
  or external vendor systems under any circumstances.
- Edge Agent must never transmit raw user behavioral data to any
  third-party inference endpoint.

23.5 Versioning All Knowledge Artifacts

All of the following are versioned and hash-tracked:
- SKILL.md files (skill_version in frontmatter).
- DESIGN.md (design_version referenced in Outcomes).
- CORE_VALUES.yml.
- DOMAIN_BOUNDARIES.json.
- Concept Graph snapshots (graph_snapshot_id).
- Benchmark baselines (tied to aria_version commit hash).
- SLM binaries (version in app bundle metadata).

23.6 Fail Loudly, Never Silently

All gate failures, budget exhaustions, benchmark regressions,
security breaches, attestation failures, and FIM alerts must:
- generate explicit, logged events,
- surface to the relevant human stakeholder,
- block dependent work until resolved.
Silent degradation is never acceptable.

23.7 Every Agent is Auditable

Every consequential action taken by any agent must be:
- traceable to a specific agent identity (cryptographic signature),
- linked to a specific ticket or session,
- reproducible via the State-Space Replay Engine,
- exportable via /aria export-audit-trail.

If an action cannot be audited, it should not be taken autonomously.
